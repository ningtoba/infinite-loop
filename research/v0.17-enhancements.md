# v0.17.x Enhancement Opportunities for infinite-loop (v14.0.0)

**Research date:** 2026-06-26
**Hermes target version:** v0.17.0 (v2026.6.19) — "The Reach Release"
**Infinite-loop target version:** v14.0.0
**Sources:** GitHub releases (NousResearch/hermes-agent), hermes-agent.nousresearch.com/docs, releasebot.io, ycptech.com command reference, ycptech.com command reference

---

## Overview

Hermes Agent v0.17.0 shipped ~1,475 commits across ~800 merged PRs since v0.16.0. The release covers new tooling, deeper delegation, cost optimizations, better memory, and automation blueprints. The infinite-loop skill at v14.0.0 already covers many features but has clear gaps where v0.17.x capabilities could be leveraged.

Below are **5 ranked enhancement opportunities**, ordered by estimated impact on the infinite-loop's reliability, efficiency, and capability.

---

## Enhancement #1 (HIGH IMPACT): Leverage `execute_code` for iteration logic within spawned sessions

**What's in v0.17.x:**
- `execute_code` tool — programmatic Python execution with RPC access to Hermes tools (web_search, web_extract, read_file, write_file, search_files, patch, terminal)
- Intermediate tool results never enter the context window — only `print()` output is returned
- 50 tool calls per execution ceiling, 300s timeout
- Two modes: `project` (uses active venv, session CWD) and `strict` (isolated temp dir)

**How infinite-loop could use it:**
Instead of the daemon orchestrating every individual tool call through a spawned Hermes session's `delegate_task`, the spawned session could use `execute_code` to batch multi-step data processing, file refactoring, or bulk search-and-replace into a single turn. This would:

1. **Reduce token consumption dramatically** — intermediate tool results stay out of the agent's context
2. **Speed up bulk operations** — loops, conditionals, and data transformations run as native Python instead of turn-by-turn tool calls
3. **Improve reliability** — atomic operations reduce the risk of mid-workflow interruption
4. **Enable new patterns** — spawned sessions could do bulk search_files + read_file + patch in a single `execute_code` call, exactly what infinite-loop's goals-file mode needs for "fix 50 lint errors"

**Implementation approach:**
- Add an `--enable-code-execution` flag to pass `code_execution` toolset to spawned sessions
- Document in SKILL.md that spawned sessions can use `execute_code` for bulk operations
- Compatible with both subprocess and library (`--use-library`) modes
- The `execute_code` tool is available on Linux/macOS (both supported platforms)

## Enhancement #2 (HIGH IMPACT): Background subagents via `delegate_task(background=true)` for parallel workstreams

**What's new in v0.17.0:**
- `delegate_task(background=true)` — dispatches a subagent that runs asynchronously and returns a handle immediately
- The parent keeps working while the background child churns
- Full result re-enters the conversation as a new turn when it finishes
- Previously, `delegate_task` was synchronous (blocked the parent until completion)

**How infinite-loop could use it:**
Currently, infinite-loop's spawned sessions either run one goal at a time or use the `--workers` flag for sequential/concurrent workers. With `delegate_task(background=true)`:

1. **An orchestrator spawned session** could fire off multiple research/investigation subagents in the background while continuing to work on other tasks
2. **Long-running goals** (e.g., "fix all lint errors in src/") could delegate separate directories to parallel background agents, then synthesize results at the end
3. **The daemon itself** (library mode) could spawn background subagents for fire-and-forget tasks (monitoring, periodic cleanup) without blocking the iteration loop

**Caveat:** Background subagents require the spawned session to be the one calling `delegate_task(background=true)`. The infinite-loop daemon orchestrates spawned Hermes sessions — the background capability is most useful at the **spawned session level**, not the daemon level. Still, this is a significant pattern the skill should document and leverage.

## Enhancement #3 (MEDIUM-HIGH IMPACT): In-session `/goal` persistent goals for spawned sessions

**What's in v0.17.x:**
- `/goal <text>` — set a standing objective that survives across turns
- After every turn, a lightweight judge model checks if the goal is achieved
- Supports completion contracts (`verify:`, `constraints:`, `boundaries:`), `/subgoal`, `/goal pause/resume/clear`
- Judge uses the `goal_judge` auxiliary model (configurable to a cheap model)
- Turn budget defaults to 20, configurable via `goals.max_turns`
- Persists in SessionDB.state_meta — survives `/resume`

**How infinite-loop could use it:**
Currently, infinite-loop daemon orchestrates iteration at the **daemon level** — it spawns a session, waits for it to complete, checks output, and decides whether to continue. v0.17's `/goal` essentially implements an **in-session iteration loop** that is more efficient because:

1. **No daemon round-trip** — the spawned session self-iterates using `/goal`, requiring the daemon only to set the goal and collect the final result
2. **Cheaper** — the judge model (cheap aux model) decides continuation instead of the daemon parsing JSON output and deciding
3. **More robust** — the goal contract mechanism with `verify:`, `constraints:`, and `stop_when:` criteria is more structured than free-text output parsing
4. **Subgoals** — spawned sessions can accumulate additional acceptance criteria via `/subgoal` mid-loop

**Implementation approach:**
- Add a `--use-goal-loop` flag that passes the iteration goal as a `/goal <text>` to the spawned session instead of the current free-text goal prompt
- The daemon sets the goal, starts the session, and lets the session self-iterate internally until the judge marks it done
- Daemon collects the final summary when the goal loop terminates
- Compatible with all existing flags (notifications, callbacks, webhooks, dashboard)
- The `/goal` command must be passed via `hermes chat -q` input or `AIAgent.run_conversation()` — not all invocation methods support it directly; library mode may need a prompt injection approach

## Enhancement #4 (MEDIUM IMPACT): Preload `execute_code` + `delegate_task(background=true)` capabilities in SKILL.md and spawned session prompts

**What's in v0.17.x:**
- Both `execute_code` and `delegate_task(background=true)` are Hermes tools the agent can discover autonomously
- But the agent is more likely to use them if the skill prompt explicitly mentions them as available patterns
- The infinite-loop SKILL.md already documents many flags and features — but does **not** yet document `execute_code` or `background` delegation

**How infinite-loop could use it:**
The spawned Hermes sessions discover their toolset at startup. When the infinite-loop spawns sessions with `--toolsets "terminal,file,delegation,code_execution"`, the agent inside will see `execute_code` as an available tool. However:

1. **The skill prompt should teach the pattern** — "When performing bulk file operations (search_files + read_file + patch across multiple files), use `execute_code` to batch them into a single turn"
2. **Background delegation should be documented** — "For parallel research tasks, consider `delegate_task(background=true)` to run sub-investigations concurrently"
3. **Agent behavior improvement** — Without explicit prompting, the agent may fall back to sequential single-call patterns that are much more expensive

**Implementation approach:**
- Update `_build_delegation_prompt()` in launch-loop.py to include guidance on `execute_code` and `delegate_task(background=true)` when relevant toolsets are enabled
- Update SKILL.md with best-practice examples for spawned sessions
- Add a note in the strategy prompt about `execute_code`'s project mode (uses same venv/CWD as terminal())

## Enhancement #5 (LOW-MEDIUM IMPACT): Atomic memory operations and SRE-style event hooks for preflight/post-iteration

**What's in v0.17.x:**
- **Memory atomic batch operations** (`memory(operations=[add/replace/remove])`) — the model can free space and add entries in a single call
- **Gateway event hooks** — `HOOK.yaml` + `handler.py` in `~/.hermes/hooks/` — fire on `agent:start`, `agent:end`, `agent:step`, `session:start`, `session:end`, `command:*`
- **Plugin hooks** — `pre_tool_call`, `post_tool_call`, `pre_llm_call`, `post_llm_call`, `on_session_start`, `on_session_end`

**How infinite-loop could use it:**
1. **Memory ops in spawned sessions** — Spawned sessions could use atomic batch memory operations to persist learned information (e.g., "which files I modified", "which errors I encountered") in a single call instead of multiple fragile add operations. This would make the spawned session's self-reporting more reliable.
2. **Event hooks for monitoring** — The infinite-loop daemon (when using library mode with AIAgent) could register plugin hooks for `post_tool_call` to log all tool activity, or `pre_llm_call` to inject context before each LLM call. This would be a lighter-weight alternative to the existing heartbeat system for detecting a hung session.
3. **Gateway hooks for webhook delivery** — If infinite-loop is used through a gateway, gateway event hooks could trigger the `--callback-url` or `--webhook-port` notifications natively instead of the current custom HTTP callback mechanism.

**Implementation approach:**
- Explore whether spawned sessions can use `memory(operations=[...])` in their output to pass structured results back to the daemon
- Add an optional `--use-memory-batch` flag that instructs spawned sessions to batch their memory writes
- Investigate hook-based preflight checks as an alternative to the current subprocess-based heartbeat system

## Additional Observations — Not Enhancements, but Relevant Context

### Already covered by v14.0.0
- **Session persistence, resume, checkpoints, worktrees** — infinite-loop already uses all these via flags
- **Notifications** — Pushbullet and ntfy already integrated
- **Dashboard** — v3 SSE already covered
- **Delegation** — the daemon already spawns sessions with `delegate_task` toolset
- **Curator** — infinite-loop doesn't need curator changes; it manages its own learning files

### v0.17.0 features NOT relevant to infinite-loop
- **New messaging platforms** (iMessage, Raft, WhatsApp Business API) — infinite-loop is CLI/daemon-based
- **Desktop app improvements** — keyboard shortcuts, themes, window management — not applicable
- **Skills Hub rehaul** — infinite-loop already manages its own skills
- **Image editing** (image-to-image) — not in scope
- **Automation Blueprints** — infinite-loop already has its own cron-like scheduling and goal file management

### v0.16.x features infinite-loop should already support (verify)
- `/undo` — spawned sessions may benefit from undo capability; verify whether infinite-loop's spawned sessions have `/undo` accessible
- `hermes chat -q` with `--toolsets` — infinite-loop already passes toolsets
- Session search FTS5 — may be useful for spawned sessions doing `session_search`

## Summary of Recommendations

| # | Feature | Impact | Effort | New Flags |
|---|---------|--------|--------|-----------|
| 1 | `execute_code` for bulk operations | 🟢 High | Low (documentation + toolset flag) | `--enable-code-execution` |
| 2 | `delegate_task(background=true)` for parallel sub-work | 🟢 High | Medium (prompt design + testing) | Document pattern in SKILL.md |
| 3 | In-session `/goal` for self-iterating sessions | 🟡 Med-High | Medium (flag + prompt change) | `--use-goal-loop` |
| 4 | Teach patterns via SKILL.md and prompts | 🟡 Medium | Low (documentation only) | None needed |
| 5 | Memory batch ops + event hooks | 🔵 Low-Med | Medium (implementation) | `--use-memory-batch`, hooks integration |

**Top recommendation for next iteration:** Add `--enable-code-execution` flag (Enhancement #1) — lowest effort ($), highest impact on token efficiency and speed of bulk operations. This is a pure additive change with zero risk to existing functionality.

**Runner-up:** Update SKILL.md and `_build_delegation_prompt()` to teach spawned sessions about `execute_code` and `delegate_task(background=true)` patterns (Enhancements #2 + #4 combined). No code changes needed, only documentation/prompt text updates.
