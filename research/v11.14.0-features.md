## Implemented Features (v11.14.0)

### Feature 1: Safe Mode (`--safe-mode`)
**Status: IMPLEMENTED**

Passes `--safe-mode` to spawned sessions, disabling ALL customizations:
user config, AGENTS.md/memory injection, plugins, and MCP servers.
Implies both `--ignore-user-config` and `--ignore-rules`. Perfect for
troubleshooting and isolating whether a problem comes from the setup
or from Hermes itself. Subprocess mode only; skipped in library mode
with a log note.

**Files modified:**
- `scripts/launch-loop.py` — Added `--safe-mode` argparser flag, passed
  through to `spawn_delegation_session()` and `run_loop()`, forwarded to
  cmd in subprocess mode. Skipped in library mode with log note.
- `SKILL.md` — Documented in changelog, options table, and pitfalls.
- `scripts/run-loop.sh` — Added flag forwarding.
- `references/cross-iteration-context.md` — Version bump.

### Feature 2: Auto-Accept Hooks (`--accept-hooks`)
**Status: IMPLEMENTED**

Passes `--accept-hooks` to spawned sessions, auto-approving any unseen
shell hooks declared in config.yaml without a TTY prompt. Useful for
CI/CD and automated pipelines where interactive prompts would block
execution. Subprocess mode only.

Files modified: Same as Feature 1.

### Feature 3: Git Worktree Mode (`--worktree`)
**Status: IMPLEMENTED**

Passes `--worktree` to spawned sessions, running them in an isolated
git worktree. Useful for parallel agents working on the same repo
without interfering with each other's working state. Subprocess mode
only.

Files modified: Same as Feature 1.

### Feature 4: Continue Session (`--continue`)
**Status: IMPLEMENTED**

Passes `--continue` to spawned sessions, resuming the most recent
session. An alternative to `--resume` that doesn't require a specific
session ID. Subprocess mode only.

Files modified: Same as Feature 1.

## Implementation Details

### Changes to `spawn_delegation_session()`:
- 4 new parameters: `safe_mode`, `accept_hooks`, `worktree`, `continue_session`
- Subprocess mode: `--safe-mode`, `--accept-hooks`, `--worktree`, `--continue`
  appended to cmd
- Library mode: silent skip with log note (no AIAgent equivalent)

### Changes to `run_loop()`:
- 4 new parameters threaded through
- Passed to both worker-path and single-execution-path spawn calls

### Changes to `main()`:
- 4 argparser entries: `--safe-mode` (store_true), `--accept-hooks`
  (store_true), `--worktree` (store_true), `--continue` (store_true,
  dest=continue_session)
- Logged in startup banner
- Wired into `run_loop()` call

## Future Directions (v12.0.0 and beyond)

With all pass-through flags now forwarded, future versions should focus on
structural improvements:

### v12.0.0 Candidates

1. **Concurrent worker improvements** — The multi-worker path auto-disables
   `--use-library`. Could add per-worker library mode with proper GIL
   management (multiprocessing instead of threading).

2. **Webhook v2 with bidirectional control** — WebSocket push for real-time
   ledger updates, plus endpoints for `/control/goal` (change goal mid-loop),
   `/control/reload` (reload config), `/control/restart` (restart the loop).

3. **Automatic error recovery** — When an iteration fails with a known error
   pattern (network, timeout), auto-adjust cooldown, retry with different
   strategy (library vs subprocess), or automatically reduce parallel workers.

4. **Ledger pruning with archiving to S3/disk** — Auto-archive old iterations
   to compressed JSONL files with configurable retention policy.

5. **Multi-profile goals** — Each goal from the goals file could specify a
   profile/model/provider tuple, enabling heterogeneous spawned sessions.

6. **Dashboard v3 (React/WebSocket)** — Real-time updates via WebSocket from
   the daemon thread to a modern dashboard with charts and live logs.

7. **Session self-repair** — When spawned sessions fail, analyze the error and
   re-spawn with adjusted parameters (shorter context, fewer tools, longer
   timeout) based on the error classification.
