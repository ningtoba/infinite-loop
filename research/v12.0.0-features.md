## v12.0.0 Candidate Features (Prioritized)

### P0: Concurrent Library Mode (`--use-library` with workers)

**Problem:** `--use-library` (AIAgent in-process) is incompatible with
`--workers > 1`. The daemon silently falls back to subprocess mode with
a log warning. This means the most efficient execution mode (library mode)
can't benefit from parallel workers.

**Root Cause Analysis:**
- `spawn_delegation_session()` at line 2500 checks `worker_id is None` 
  and only enters library mode when that's true
- Multi-worker path (line 3384) uses `ThreadPoolExecutor` and explicitly
  sets `use_library=False` at line 3413
- AIAgent is NOT thread-safe ("Create one AIAgent per thread or task.
  Never share an instance across concurrent calls.")
  - But AIAgent **IS** fork-safe via multiprocessing (per batch_runner.py)
  - `batch_runner.py` (line 918) uses `Pool(processes=N)` with
    `multiprocessing` to run parallel AIAgent conversations

**Solution:** Replace `ThreadPoolExecutor` with `multiprocessing.Pool` for
the multi-worker path when `--use-library` is active. Each worker process
creates a fresh AIAgent instance with its own config.

**Implementation Plan:**
1. Import `multiprocessing.Pool` (stdlib — already available)
2. Create a worker function `_library_worker()` at module level (must be
   picklable) that:
   - Takes a serializable config dict (goal, context, toolsets, model, etc.)
   - Creates a fresh AIAgent instance
   - Calls run_conversation()
   - Returns the result dict
3. In `run_loop()`, when `workers > 1` and `use_library`:
   - Serialize per-worker config (goal from goals_file cycling)
   - Submit to `Pool(processes=workers).map()` (simpler than imap)
   - Collect results
4. **Key concern:** The `prompt` is constructed in `spawn_delegation_session()`
   via `_build_delegation_prompt()`. For multiprocessing, the prompt should
   be constructed in the main process and passed to workers.
5. After all workers complete, process results the same way as the current
   ThreadPoolExecutor path (merge contexts, calculate errors)

**Files to modify:**
- `scripts/launch-loop.py` — Replace thread pool with process pool in
  the multi-worker library path (~lines 3384-3447)

**Edge cases:**
- Workers that crash (segfault, OOM) — `Pool.map` raises `BrokenProcessPool`
  — catch and treat as error
- Pickling errors — the config dict must contain only basic types (str, int,
  bool, None). Complex objects like callbacks or open file handles will fail.
  Use `copyreg` or lambda-free structures.
- Resource tracking from `/proc` — each worker is a separate process, so
  /proc stats are per-worker. Aggregate in the parent.
- `AIAgent` import — each worker process imports independently (slow first
  time but cached per process)
- Memory overhead — N workers = N x AIAgent memory. Monitor via
  `--max-output-chars` capping and `--keep-iterations`.

### P1: Automatic Error Recovery

**Problem:** When an iteration fails, the current code only does:
- Exponential backoff (`retry_delay * min(consecutive_errors, 5)`)
- Optional retry of the same iteration (`--max-retries`)
- Error classification (timeout, network, schema, unknown)

It does NOT adapt the daemon's behavior based on error patterns.

**Current Error Handling (lines 3853-3890):**
```python
if cooldown > 0 or cooldown_mode == "adaptive":
    effective_cooldown = calc_adaptive_cooldown(avg_dur)
    _sleep_with_shutdown_check(effective_cooldown, sentinel_path)

# Backoff
consecutive_errors = 0 if not combined_error else consecutive_errors + 1
if combined_error and retry_delay > 0 and consecutive_errors > 0:
    delay = retry_delay * min(consecutive_errors, 5)
    time.sleep(delay)
```

**Proposed Recovery Strategies:**

| Error Type | Strategy |
|------------|----------|
| `timeout` | Increase session_timeout by 50%, or switch from library to subprocess mode (subprocess has independent timeout) |
| `network` | Increase cooldown by 300%, retry with exponential backoff |
| `schema` | Log full schema error, skip output validation for next N iterations |
| `unknown` | Reduce parallel workers to 1, switch to subprocess mode |

**Implementation Plan:**
1. Add `_adapt_to_error(state, error_type, consecutive_errors)` function
2. Track per-error-type counts in the ledger (e.g., `error_counts.timeout = 3`)
3. After 3 consecutive timeouts: auto-double session_timeout
4. After 2 consecutive network errors: increase cooldown to 120s minimum
5. After 5 consecutive unknown errors: force subprocess mode for 10 iterations
6. On recovery (successful iteration after errors): gradually reduce mitigations

**Files to modify:**
- `scripts/launch-loop.py` — Add error recovery function, modify
  post-iteration error handling section (~lines 3853-3920)

### P1: In-Process Ledger Archiving

**Problem:** The current archiving requires an external shell script
(`archive-state.sh`). The `--keep-iterations` feature trims the in-memory
ledger but doesn't save the archived entries anywhere — data is lost.

**Current State:**
- `--keep-iterations N` (line 3850 area): trims ledger to last N entries
  when it reaches 2N. Does NOT archive the trimmed entries.
- `archive-state.sh`: wraps the process in a shell script, reads from
  ledger state file, writes JSONL archive files.
- The trimmed data is permanently lost — no way to recover old iterations.

**Proposed Solution:** Integrated auto-archiving that writes compressed
JSONL files to `~/.hermes/infinite-loop-archives/` whenever the ledger
is trimmed, with configurable retention.

**Implementation Plan:**
1. Add `--archive-dir DIR` flag (default: `~/.hermes/infinite-loop-archives/`)
2. In the ledger-trimming code (around line 3850), before removing old
   entries, serialize them to a JSONL archive file:
   - File name: `iterations-{YYYYMMDD}-{seq}.jsonl.gz`
   - One JSON object per line (standard JSONL format)
   - Gzip-compressed (stdlib `gzip`)
3. Add auto-cleanup of archives older than `--archive-retention N` days
   (default: 30)
4. Add `--archive-max-size MB` (default: 100) — when archives exceed this,
   compress older ones or alert
5. Update `replay-ledger.sh` to read gzip JSONL natively (it already
   supports `.jsonl.gz` per the SKILL.md docs)
6. Add metadata header to each archive file:
   ```json
   {"_meta": {"version": 11, "archived_at": "2026-06-26T02:30:00Z", ...}}
   ```

**Files to modify:**
- `scripts/launch-loop.py` — Add `_archive_iterations()`, modify
  ledger-trimming code
- `scripts/replay-ledger.sh` — Verify gzip support
- `scripts/archive-state.sh` — Can remain for manual archiving

### P2: Multi-Profile Goals File

**Problem:** When using `--goals-file`, all goals are executed with the same
profile/model/provider. There's no way to use different profiles for
different goals in the same batch run.

**Proposed Format:** Extended goals file where each line can optionally
specify profile, model, and provider as pipe-separated fields:
```
Fix type errors in src/auth/|work|claude-sonnet-4|
Refactor database layer|dev|gpt-4o|
Update documentation|||openai
```

**Implementation:**
1. In `load_goals()` (line ~3290 area), detect pipe-separated format
2. Parse each line as: `goal|profile|model|provider`
3. Store as a tuple `(goal, profile_override, model_override, provider_override)`
4. In the spawn loop, use per-goal profile/model/provider overrides
5. Backward compatible — plain lines (no pipes) use the daemon defaults

**Files to modify:**
- `scripts/launch-loop.py` — Modify `load_goals()` and spawn loop

## Design Decisions

### Why multiprocessing.Pool over ThreadPoolExecutor for workers

| Aspect | ThreadPoolExecutor (current) | multiprocessing.Pool (proposed) |
|--------|------|------|
| AIAgent thread safety | ❌ Not safe, forced subprocess | ✅ Each process has own AIAgent |
| GIL impact | ⚠️ Limited by Python GIL | ✅ True parallelism |
| Memory overhead | ⚠️ Shared (high per-thread) | ⚠️ Replicated (Nx memory) |
| Import time | ✅ Once | ⚠️ N times (cached per-process) |
| Pickle overhead | ✅ None | ⚠️ Serialize config dict |
| Crash isolation | ❌ One crash kills loop | ✅ Worker crash = retry |
| Code complexity | ✅ Simple submit/result | ⚠️ Worker function + config |

### Why aixi-style error recovery style

The recovery system follows a "nudge, don't force" design:
- Start with minimal intervention (increase timeout 50%)
- Escalate gradually (reduce workers, switch mode)
- Auto-recover on success (reset mitigations)
- Never disable safety features permanently

### Archive file format

JSONL.gz was chosen over SQLite or plain JSON for:
- Human-readable (gunzip | head)
- Append-friendly (one line per iteration)
- Compressible (gzip typically 10:1 for JSON text)
- Replayable (replay-ledger.sh already supports it)
- Greppable (zgrep works on compressed JSONL)
