### P1: Output Progress Classification

**Problem:** The daemon stores iteration summaries but doesn't classify them.
Each iteration is either "success" or "error" — there's no concept of "partial
progress" vs "stuck" vs "completed the task".

**Solution:** Add `_classify_progress()` function that analyzes each iteration's
summary and git diff to classify it:

```python
def _classify_progress(summary: str, git_diff_stat: str) -> str:
    """Classify iteration progress.

    Returns one of: 'completed', 'progress', 'partial', 'stuck', 'regression'
    """
    # Check for completion keywords
    if any(kw in summary.lower() for kw in ['completed', 'finished', 'all done',
                                             'all tasks', 'all fixes']):
        return 'completed'
    # Check for regression (errors after previous success)
    if 'error' in summary.lower() and not git_diff_stat:
        return 'regression'
    # Check for stuck (no changes, repetitive summary)
    if not git_diff_stat:
        return 'stuck' if len(summary) < 20 else 'partial'
    # Has changes = making progress
    return 'progress'
```

Store in ledger per-iteration: `"progress": "partial"`.

**Benefits:**
- Dashboard can color-code iterations (green=completed, yellow=progress, red=stuck)
- Auto-stop criteria can be tuned: e.g., stop after 3 "stuck" classifications
- Better monitoring: know when the loop is actually accomplishing things

### P2: Idempotent Goal Execution (Skip-If-Done)

**Problem:** With `--goals-file`, all goals are executed regardless of whether
they've already been completed in a previous run. If the daemon crashed after
20/30 goals, restarting re-does the first 20 goals.

**Solution:** Track per-goal completion status in the ledger using a hash of
the goal text. Before executing a goal, check if it was already completed:

```python
goals_tracked = state.setdefault("goals_completed", {})

for goal_spec in pending_goals:
    goal_hash = hashlib.md5(goal_spec.goal.encode()).hexdigest()[:16]
    if goal_hash in goals_tracked and goals_tracked[goal_hash].get("status") == "completed":
        _log(f"[SKIP] Goal already completed: {goal_spec.goal[:80]}...")
        continue
    # ... execute normally
```

Also add `--reset-goals` to clear completion tracking.

### P2: Session Self-Healing

**Problem:** When a spawned Hermes session crashes mid-iteration (network blip,
OOM, process kill), the daemon only knows it failed — it doesn't try to
recover the in-flight work.

**Current behavior:**
- `spawn_delegation_session()` raises exception → caught by retry loop
- After max_retries exhausted → iteration recorded as error
- Crash recovery (ledger pendings iteration check) only works on DAEMON restart, not mid-run

**Solution:** Add mid-iteration self-healing:

1. **Periodic heartbeat** — Spawned sessions emit a heartbeat file or signal every 30s
2. **Heartbeat expiry** — If heartbeat stops mid-iteration, kill the session and retry
3. **Checkpoint resumption** — If the session was using `--checkpoints`, resume from latest checkpoint
4. **Grace period** — Allow 60s for the session to recover before declaring failure

```python
def _monitor_heartbeat(pid: int, heartbeat_path: str, timeout: int = 60) -> bool:
    """Returns False if heartbeat stopped before session completed."""
    deadline = time.time() + timeout
    while time.time() < deadline:
        if not os.path.exists(heartbeat_path):
            return False  # Heartbeat missing
        mtime = os.path.getmtime(heartbeat_path)
        if time.time() - mtime > 30:
            return False  # Stale heartbeat (30s without update)
        time.sleep(5)
    return True  # Session completed within timeout
```

### P2: MCP-Based Control Interface

**Problem:** Currently, the daemon can only be controlled via:
- Sentinel file (`echo "stop" > /tmp/infinite-loop-stop`)
- REST API (webhook HTTP endpoints)
- Signal handler (SIGTERM/SIGINT)

This requires external scripts or manual intervention. No programmatic
control from Hermes itself.

**Solution:** Expose daemon control as MCP tools so Hermes can manage the
loop natively. Add MCP server mode:

```python
# New flag: --mcp-server (experimental)
# Starts an MCP server alongside the loop that exposes:
# - control_stop() → stop the loop
# - control_pause() → pause after current iteration
# - control_resume() → resume from pause
# - get_status() → full ledger state
# - get_iteration(n) → specific iteration record
# - set_goal(text) → override goal mid-run
# - add_goal(text) → append to goals_file queue
```

MCP tools are defined as JSON Schema, discovered via `tools/list`,
and invoked via `tools/call`. This is the same protocol Hermes uses
for its own MCP servers (Chroma, Cognee, fastCRW).

### P3: Multi-Machine Workers

**Problem:** `--workers N` only creates local parallel sessions. All workers
run on the same machine with the same resources.

**Solution:** Allow `--worker-url` to accept multiple URLs for distributed
workers across machines:

```bash
python3 scripts/launch-loop.py --goal "Fix type errors" \
  --worker-url "http://worker1:8124,http://worker2:8124,http://worker3:8124" \
  --run
```

Each iteration distributes work across available remote workers. Workers
register with the daemon via the webhook endpoint. Fallback to local if
remote is unavailable.

## Design Decisions

### Why SSE over WebSocket

- Python stdlib `http.server` supports SSE natively (just keep connection open)
- WebSocket requires external library (websockets or asyncio)
- SSE is simpler: text/event-stream with `data:` lines
- SSE auto-reconnects on connection loss (browser built-in)
- SSE is fire-and-forget — no client→server messages needed for dashboards

### Why Self-Test Over Unit Tests

- The daemon is a single-file stdlib script with no test framework
- Adding pytest + fixtures adds complexity and a new dependency
- A `--self-test` flag runs entirely in-process with mock data
- Same pattern as the daemon itself: self-contained, no external deps
- Can be extended to CI/CD: `python3 launch-loop.py --self-test --exit-on-fail`

### Progress Classification Design

The classification is intentionally simple (5 categories, keyword-based):
- **Heuristic, not ML** — no model calls needed per iteration
- **Fast** — O(n) in summary length (n < 500 chars typically)
- **Stateless** — doesn't depend on past classifications (except "regression")
- **Composable** — classification stored per-iteration for dashboard use
