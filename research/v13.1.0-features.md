## v13.1.0 Candidate Features (Prioritized)

### P0: Dashboard v3 — Server-Sent Events (SSE) Real-Time Updates

**Problem:** The current HTML dashboard (`--status-html`) auto-refreshes via
`<meta http-equiv="refresh">` every 30 seconds. This is polling-based: high
latency (up to 30s stale data), no server push, and the browser shows a
blinking refresh indicator.

**Solution:** Add a `GET /live` SSE endpoint to the existing `WebhookHandler`
and a `GET /dashboard` live-updating HTML page with JavaScript `EventSource`:

```python
class WebhookHandler:
    # New class-level list for SSE clients
    _sse_clients: list = []  # list of (request_handler, queue)
    
    def do_GET(self):
        if self.path == '/live':
            self._handle_sse()
        elif self.path == '/dashboard':
            self._handle_dashboard()
    
    def _handle_sse(self):
        self.send_response(200)
        self.send_header('Content-Type', 'text/event-stream')
        self.send_header('Cache-Control', 'no-cache')
        self.send_header('Connection', 'keep-alive')
        self.end_headers()
        # Keep connection open, read from queue
        queue = queue.SimpleQueue()
        WebhookHandler._sse_clients.append(queue)
        try:
            while True:
                data = queue.get()  # blocks
                self.wfile.write(f'event: iteration\ndata: {json.dumps(data)}\n\n'.encode())
                self.wfile.flush()
        except (BrokenPipeError, ConnectionResetError):
            pass
        finally:
            if queue in WebhookHandler._sse_clients:
                WebhookHandler._sse_clients.remove(queue)
```

**Communication mechanism:** After each iteration completes in `run_loop()`,
broadcast the iteration data to all SSE clients by iterating the class-level
list and putting data on each client's queue.

**Benefits:**
- Real-time updates (sub-second latency)
- No browser refresh indicator
- Lower server load (push vs poll)
- Full iteration data available in JS

### P1: Session Self-Healing

**Problem:** When a spawned Hermes session crashes mid-iteration (network blip,
OOM, process kill), the daemon only knows it failed — it doesn't try to
recover the in-flight work.

**Solution:** Three-layer recovery:

1. **Heartbeat mechanism** — Add heartbeat instructions to the spawned session
   prompt. The spawned session touches a heartbeat file every 30s. New flag:
   `--heartbeat-timeout` (default: 0 = disabled).

2. **Heartbeat monitoring** — A monitoring thread watches the heartbeat file
   during `_execute_iteration()`. If heartbeat stops for `--heartbeat-timeout`
   seconds, kill the child and retry.

3. **Checkpoint resumption** — If `--checkpoints` is active and the heartbeat
   expires, attempt to resume from the latest checkpoint in the spawned
   session's workdir.

**Design details:**
- Heartbeat file path: `/tmp/infinite-loop-heartbeat-{pid}.touch`
- Monitoring uses a separate thread (stdlib threading, no extra deps)
- Store heartbeat state in ledger as `heartbeat_ok` boolean per iteration

## Design Decisions

### Why SSE over WebSocket

- Python stdlib `http.server` supports SSE natively (just keep connection open)
- WebSocket requires external library (websockets or asyncio)
- SSE is simpler: text/event-stream with `data:` lines
- SSE auto-reconnects on connection loss (browser built-in)
- SSE is fire-and-forget — no client→server messages needed for dashboards

### Why Heartbeat-Based Self-Healing

- stdlib-only (`os.path.getmtime`, `threading.Thread`, `time.sleep`)
- No external signaling mechanism needed
- Simple to implement from spawned session side (just `touch` a file)
- Compatible with existing checkpoint system
- Heartbeat timeout is configurable and disabled by default (opt-in)
