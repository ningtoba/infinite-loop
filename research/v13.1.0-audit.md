### 3.2 `--shutdown-sentinel` Details

The `--shutdown-sentinel` flag lets the user override the default sentinel file path (`/tmp/infinite-loop-stop`). This is a niche feature but should still be forwardable through the shell wrapper for completeness.

### 3.3 `--run` Details

`--run` is intentionally not in run-loop.sh because the shell script always passes `--run`. This is a design choice, not a real gap. But documenting this in the script would reduce confusion for future maintainers.

## 7. New Enhancement Opportunities

These go beyond existing research coverage and represent creative additions for v13.1.0.

### 7.1 P0: Webhook Notification Payload Enrichment

**Current state:** The webhook HTTP POST (`--http-callback URL` and notification commands) sends a JSON payload with basic iteration data: `n`, `summary`, `error`, `duration_seconds`, `classification`, etc.

**Enhancement:** Enrich the JSON payload to include:
- `iteration` block with full current iteration data (not just summary)
- `state` block with daemon-level metrics (total duration, success/error counts, ETA)
- `stats` block with rolling averages, consecutive successes/errors
- `resources` block with CPU/memory tracking data
- `system` info (hostname, pid, workdir)
- URL to the live dashboard (if `--status-html` is set)

**Benefit:** External monitoring systems (Grafana, custom dashboards, CI/CD pipelines, home automation) get richer data without needing to poll the ledger file or `/api/status` endpoint separately.

**Implementation effort:** Low — ~30 lines in `_handle_notifications()` / `_build_iteration_record()`.

### 7.2 P1: Error Messaging in Dashboard

**Current state:** The `--status-html` dashboard shows errors as highlighted rows with the error string. There's no structured error display, no error aggregation, no per-type counters visible.

**Enhancement:** Add an error panel to the dashboard:
- Error summary card: "3 consecutive errors (2 timeout, 1 network)"
- Active mitigations display: "Session timeout extended to 900s, workers reduced to 1"
- Error timeline: mini bar chart or table showing error type distribution
- Link to `/api/status` for raw data

**Implementation effort:** Medium — extend `_STATUS_HTML_TPL` and `_generate_status_html()` to include error stats from `state["error_type_counts"]` and `state["mitigations"]`.

### 7.3 P1: Performance Metrics in Dashboard

**Current state:** The dashboard shows iteration count, duration, and ETA. No average turn counts or token usage.

**Enhancement:** When available (library mode returns token usage data; subprocess mode could estimate):
- Average turns per iteration
- Token usage estimate (prompt + completion tokens per iteration)
- Cost estimate (if using library mode with price-per-token config)
- Dashboard card: "Avg 47 turns/iter, ~12K tokens/iter"
- Enable token-aware auto-stopping: stop after estimated budget exhausted

**Implementation effort:** Medium — requires token tracking in the ledger `state["stats"]["total_tokens"]`, new dashboard cards.

### 7.4 P1: Spawned Session Prompt Size Reduction via Skill Preloading

**Current state:** The daemon builds large prompts with context injection, failure learning context, cross-iteration context, tool instructions, etc. For library mode, the `_build_delegation_prompt()` produces prompts that can exceed 10K characters.

**Enhancement:** Move repetitive instruction content into a hermes skill that is preloaded (`-s` flag) into spawned sessions instead of embedding it in the prompt. This reduces prompt tokens and makes spawned sessions faster to start.

**How it works:**
1. Create a small skill `infinite-loop-spawned` in `~/.hermes/skills/` with the standard instruction block
2. The daemon detects if the skill is available; if so, passes `-s infinite-loop-spawned` and removes redundant instructions from the prompt
3. Fall back to inline instructions if the skill isn't available

**Benefit:** Prompt size reduction of 30–50% (~2–5K fewer tokens per iteration), faster session startup, cleaner separation of concerns.

**Implementation effort:** Medium — new skill file, prompt-building logic changes, detection/fallback logic.

### 7.5 P1: Goal Completion Dashboard — Progress Visualization

**Current state:** With `--goals-file`, the daemon processes goals sequentially or in parallel. The dashboard shows "Total: N goals" but no per-goal completion status.

**Enhancement:** Dashboard goals panel:
- Goal progress bar: "12/30 goals complete (40%)"
- Per-goal status list: each goal with ✓ or ✗
- Time-per-goal estimates based on rolling average
- ETA for goals-file completion

**Implementation effort:** Medium — needs `state["goals_completed"]` tracking, `_STATUS_HTML_TPL` updates.

### 7.6 P2: Smart Goal Ordering in Goals File

**Current state:** Goals from `--goals-file` are processed sequentially. No dependency ordering.

**Enhancement:** Support optional dependency annotations in the goals file:
```
goal_text [depends:2,5]        # Depends on goals 2 and 5 completing first
goal_text [after:phase1]       # Depends on 'phase1' tag completing
```

The daemon topologically sorts goals before executing, skipping unblocked goals.

**Implementation effort:** Medium-high — requires `_resolve_goal_dependencies()`, dependency graph, blocked-goal waiting logic, circular dependency detection.

### 7.7 P2: Preflight Check Enhancement — Hermes Version Check

**Current state:** Preflight checks verify hermes is on PATH and that basic files exist. No version compatibility check.

**Enhancement:** Add a preflight check that runs `hermes --version` and verifies the installed Hermes version is compatible with the daemon's features. Emit a warning if the version is too old for certain flags.

**Example:**
```
✓ hermes found: Hermes Agent v0.17.0
  hermes version supports spawned flags: OK (v0.17.0 >= v0.15.0)
  hermes version supports --profile flag: OK
```

**Implementation effort:** Low — ~20 lines in `run_preflight_checks()`.

### 7.8 P3: `--quiet` Mode for run-loop.sh

**Current state:** `run-loop.sh` always prints its ASCII banner (13 lines) + startup messages.

**Enhancement:** Add `--quiet` / `-q` flag to suppress banner output for CI/CD pipelines and scripted use:
```bash
bash scripts/run-loop.sh --goal "..." --quiet
```

**Implementation effort:** Very low — ~5 lines.

### 7.9 P3: JSON Output Mode for run-loop.sh

**Current state:** `run-loop.sh` outputs human-readable text. Parsing the output programmatically is brittle.

**Enhancement:** Add `--json-output` flag that outputs JSON-only startup info:
```json
{"version": "13.0.0", "pid": 12345, "goal": "...", "ledger": "/tmp/..."}
```

**Implementation effort:** Low — ~10 lines in run-loop.sh.

### 7.10 P3: Config File Schema Validation

**Current state:** `--config PATH` loads a JSON file but doesn't validate it against an expected schema. Malformed config files may produce cryptic errors.

**Enhancement:** Use the existing `validate_json_output()` (or a new `validate_config()`) to check loaded configs against the expected argument schema. Report clear errors on unknown keys or invalid types.

**Implementation effort:** Low — ~20 lines, reuses existing validation infrastructure.

### 7.11 P4: Spawned Session Image Attachment Support

**Current state:** No way to attach images to spawned sessions.

**Enhancement:** Add `--spawn-image PATH` flag that passes `--image PATH` to `hermes chat -q`. Useful for spawned sessions that need visual context (e.g., "Fix this UI bug" with a screenshot).

**Implementation effort:** Low — ~5 lines in launch-loop.py + run-loop.sh.

### 7.12 P4: `hermes insights` Integration

**Current state:** The daemon has no integration with Hermes' analytics/insights features.

**Enhancement:** After the daemon completes (or periodically), run `hermes insights` to log usage data. Store insight snippets in the ledger for post-run analysis.

**Implementation effort:** Low — ~15 lines.
