### P1: Dashboard Error Panel

**Problem:** The SSE dashboard shows iteration data but doesn't surface error patterns.
When 3 consecutive timeouts occur, the operator has to dig through the console log.

**Solution:** Add an error summary card to the dashboard HTML:
- Error type counts (timeout=5, parse=2, heartbeat=1)
- Active mitigations (session_timeout=900s, workers=1)
- Error timeline (recent failures in reverse chronological)

**Files:** scripts/launch-loop.py (_SSE_DASHBOARD_HTML_TPL template, _build_sse_payload function)
**Effort:** Medium (~40 lines in HTML template + payload builder)
**Impact:** High — better operational visibility

### P1: Performance Metrics in Dashboard

**Problem:** The SSE dashboard shows iteration count and duration but no average
turn counts or token usage.

**Solution:** When available (library mode returns token usage data):
- Average turns per iteration
- Token usage estimate (prompt + completion tokens)
- Cost estimate (when library mode with price-per-token config)
- Dashboard card: "Avg 47 turns/iter, ~12K tokens/iter"

**Files:** scripts/launch-loop.py (_SSE_DASHBOARD_HTML_TPL, _build_sse_payload)
**Effort:** Medium (~30 lines in templates + payload builder)
**Impact:** High — data-driven insights

### P1: Goal Completion Dashboard Visualization

**Problem:** With `--goals-file`, the dashboard shows "Total: N goals" but no per-goal
completion status. Users have to check the console log.

**Solution:** Dashboard goals panel:
- Goal progress bar: "12/30 goals complete (40%)"
- Per-goal status list with ✓/✗ indicators
- Time-per-goal estimates based on rolling average
- ETA for goals-file completion

**Files:** scripts/launch-loop.py (_SSE_DASHBOARD_HTML_TPL, _build_sse_payload,
state["goals_completed"] tracking)
**Effort:** Medium (~50 lines)
**Impact:** Medium — better progress tracking for batch processing

### P2: Spawned Session Prompt Optimization via Skill Preloading

**Problem:** `_build_delegation_prompt()` produces prompts that can exceed 10K characters.
The repetitive instruction content inflates token usage for every iteration.

**Solution:** Create a small skill `infinite-loop-spawned` in `~/.hermes/skills/` with
the standard instruction block. When `--skills` is used with the skill name, the
daemon detects it and removes redundant instructions from the prompt. Fall back to
inline instructions if the skill isn't available.

**Files:** scripts/launch-loop.py (prompt-building logic), new skill file
**Effort:** Medium (~40 lines + new skill file)
**Impact:** High — reduces prompt tokens by 30-50%, faster session startup

### P2: Goal Hash Tracking (--track-goals Persistence)

**Problem:** With `--track-goals`, the daemon persists completed goal text as plain
strings. If a goal text changes slightly between runs, it may not match.

**Solution:** Use `hashlib.md5` of goal text + iteration number. Store goal hashes
in `goals_completed` as a dict mapping hash → {goal, iteration, timestamp}. This
ensures goals are uniquely identified regardless of text variations.

**Files:** scripts/launch-loop.py (_cycle_goal function), scripts/session-self-loop.py
**Effort:** Low (~15 lines)
**Impact:** Medium — robustness improvement

### P2: Webhook Payload Enrichment

**Problem:** Current `--http-callback` POSTs the raw iteration record. No metadata
like iteration number, total iterations, or ETA.

**Enhancement:** Enrich the notification payload with:
- `iteration_number`, `total_iterations` at the top level
- `progress_percentage`, `eta_seconds`
- Worker breakdown in multi-worker mode

**Files:** scripts/launch-loop.py (_handle_notifications)
**Effort:** Very Low (~10 lines)
**Impact:** Medium — better external monitoring

### P3: --quiet Mode for run-loop.sh

in CI/CD pipelines and scripted use.

**Solution:** Add `--quiet` / `-q` flag to suppress banner output. The JSON output
at the end is still printed.

**Files:** scripts/run-loop.sh
**Effort:** Very Low (~5 lines)
**Impact:** Low — CI/CD convenience

### P3: Config File Schema Validation

**Problem:** `--config PATH` loads a JSON file but doesn't validate it against an
expected schema. Malformed config files may produce cryptic errors.

**Solution:** Use `validate_json_output()` to check loaded configs against the
expected argument schema. Report clear errors on unknown keys or invalid types.

**Files:** scripts/launch-loop.py
**Effort:** Low (~20 lines, reuses existing validation)
**Impact:** Medium — robustness
