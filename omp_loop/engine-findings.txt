## New findings — NOT in ENGINEERING_BACKLOG.md

### 1. Duplicated content_block_stop handler in _execute_task (medium)
The _execute_task function in loop.py has TWO identical blocks handling `content_block_stop` events (lines ~191-209 and ~210-227). Both check if delta.type == "tool_result" and render it via _term(). This wastes CPU cycles and can double-display tool results in terminal output.
- File: omp_loop/loop.py lines ~191-227
- Fix: Remove the second handler block

### 2. _build_dashboard_html hardcodes /tmp path (low)
The HTML dashboard builder hardcodes `/tmp/infinite-loop-state.json` in suggested commands even when `run_loop()` is configured with a different data directory.
- File: omp_loop/loop.py, _build_dashboard_html()
- Fix: Pass the actual ledger path as a parameter

### 3. Docker detection via /.dockerenv is fragile (low)
The only Docker-aware code is in web_app/loop_manager.py, but engine-side file_utils.py has no Docker awareness.
- File: omp_loop/loop.py and related modules
- Fix: Add a centralized `is_docker()` utility

### 4. _cycle_goal uses modulus instead of tracking exhaustion (low)
When stop_at_goals_end is False, goals cycle via modulus with no cap — but the exhaustion logic only works when stop_at_goals_end is True. The modulus wrapping means an unbounded loop with goals_file cycles through goals forever regardless of the limit.
- File: omp_loop/functions.py _cycle_goal()
- Note: This is by-design for the non-stopping case, but lacks a "max_cycle" guard
